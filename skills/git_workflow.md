# Git Workflow for AI Agents

## Context
When an AI agent is working on a long-running task, operating under a `/goal`, or performing complex code refactoring. Without proper Git isolation and checkpointing, agents risk ruining the `main` branch, losing their own progress, or making it impossible for the user to review massive, monolithic changes.

## Solution
All agents MUST abide by this 4-step Git methodology:

1. **AI Traceability (Commit Prefixes)**
   - EVERY commit generated by an AI must be visually distinct.
   - Format: `[type]: 🤖 [AI] [description]`
   - Example: `feat: 🤖 [AI] implement initial database schema`

2. **Incremental Checkpointing**
   - DO NOT wait until the end of a `/goal` or a complex task to commit.
   - Commit continuously after completing any logical sub-task, test, or paragraph.
   - This prevents data loss and allows the user to rollback to a safe state if the agent hallucinates later.

3. **Smart Branching**
   - **Trivial/Minor edits** (e.g. typos, small paragraphs, formatting): commit directly to `main`.
   - **Significant/Structural edits** (e.g. refactoring, creating architecture, deep reasoning): execute `git checkout -b ai/<feature-name>`. Work there, push the branch, and request user review before merging.

4. **Git Worktree for Extreme Isolation**
   - If you need to heavily experiment with build tools, dependencies, or large-scale project overhauls, use a separate worktree.
   - Command: `git worktree add ../<worktree-name> <branch-name>`
   - This keeps the primary repository completely clean while you destroy/rebuild the experimental worktree.

## Gotchas / Warnings
- **NEVER** use `git push --force` on the main branch.
- **NEVER** hide your changes in a single massive commit called "done". The user will reject it.
- When working inside a `git worktree`, remember that your CWD needs to explicitly change to the new worktree path before running compilation commands or executing tools.
